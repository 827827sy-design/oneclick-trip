# 一键游全链路测试方案与阶段报告

更新日期：2026-07-22

## 1. 测试目标

验证用户请求能从 Vue 用户端进入 Spring Boot，经 FastAPI/LangGraph 调度模型、工具、记忆和知识库，再将会话与行程写入 MySQL，并可在 Vue 管理端按用户查看。预订仅验证安全架构和 Mock Java Backend，不验证真实支付或供应商交易。

## 2. 测试分层

### 提交前 Smoke（目标 1 分钟内）

- Agent：意图路由、复合查询、规划校验、修改、预订安全守卫、RAG 检索。
- Java：`AiAssistantServiceTest`。
- 前端：用户端和管理端 `vite build`。
- 服务：MySQL、Redis、FastAPI、Spring Boot、两个 Vite 服务端口及健康接口。

### 合并前 Full

- Agent 全量 pytest，按 `core`、`flows`、`knowledge`、`research-api` 四组并行。
- Spring Boot 全量 Maven Test。
- 两个前端生产构建。
- HTTP 真实链路：登录、异步任务、进度轮询、会话、规划、修改、长期记忆、RAG、预订守卫。
- Playwright：手机用户端和桌面管理端关键路径。

### 答辩前 Acceptance

- 使用固定演示账号和固定问题集，连续执行三轮。
- 记录规划总耗时、各阶段耗时、模型/工具失败率和方案保存成功率。
- 人工复核天气日期、路线距离、预算、景点开放时间、资料来源和前端排版。

## 5. 本轮发现并修复

1. “查询明天天气，并推荐第一次去成都的住宿区域”被误判为完整规划。
   - 原因：规则把“明天”和普通语义中的“去成都”组合成了行程信号。
   - 修复：只有明确天数、预算、出发、旅游等强信号才升级为完整规划。

2. 成都规划被 `KNOWLEDGE_RESULTS_NOT_USED` 拦截。
   - 原因：一篇新疆资料被错误标注为成都，且正文城市列表偶然含“成都”，低分脏命中触发了强制引用校验。
   - 修复：RAG 增加最低重排分 `0.48`；低分资料不进入 Agent 上下文，高分资料仍要求引用溯源。

3. “喀纳斯和禾木怎么安排更合理”被误判为完整规划并追问天数。
   - 修复：模型想升级为 `trip_plan` 但代码没有发现强规划信号时，保留为只读知识问答；已有规划的槽位补充仍可恢复到原规划意图。

## 6. 当前边界与风险

- 当前没有真实酒店/交通/门票供应商 option id，因此在线预订只能停在槽位追问；正向草稿确认链路使用 Mock Java Backend 验证。
- 完整规划约 69 秒，未超过两分钟，但应继续统计 P50/P95，并评估候选选择、格式化等简单节点切换到 Flash 模型。
- 已发布知识中仍存在广告、错误城市元数据和非官方来源。相关性门槛能防止明显误用，但不能替代后台资料治理。
- 用户端 `index.html` 标题仍是“一键游管理后台”，两个站点缺少 favicon，均不影响业务但应在答辩前修正。
- Chroma 依赖产生 Python 3.16 兼容性弃用警告，目前不影响运行。

## 7. 推荐固定用例

1. 单项：`成都明天天气怎么样？`
2. 复合：`查询成都明天天气，并推荐两个适合第一次去成都游客的住宿区域。`
3. 规划：`帮我规划成都2日游，1个人，2026年8月10日到8月11日，总预算3000元，喜欢美食和历史文化，节奏轻松。`
4. 修改：`交换第一天和第二天的安排，其他条件不变。`
5. 预订守卫：`帮我预订这份方案里的酒店。`
6. 记忆：`请记住我喜欢自然景观和高铁，不喜欢购物。`
7. RAG：`新疆北疆的喀纳斯和禾木怎么安排更合理？请给出资料来源。`
8. 失败恢复：天气服务超时、知识库无匹配、模型返回非法 JSON、硬校验两轮失败、重复确认预订。
